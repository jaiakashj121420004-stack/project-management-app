- **2026-06-21** — **Phase 9 PWA & reminders.** **SW = `registerType: 'prompt'` + `useRegisterSW`** (not `autoUpdate`): a Workbox `generateSW` precaches the app shell (`globPatterns`, `navigateFallback: index.html` for SPA/offline routes, `maximumFileSizeToCacheInBytes` lifted to 4 MB since the single bundle is ~950 kB pre-split), and a custom **PWAReloadPrompt** offers a one-tap reload so updates never blow away a mid-edit — `injectRegister` auto-detects the virtual import so there's **no double registration** (confirmed: no inline `registerSW` in built HTML). **Icons are generated, not bundled deps** — `scripts/generate-icons.mjs` hand-encodes PNGs (built-in `zlib` + a CRC32 table, per-pixel aurora gradient + SDF "A" glyph) → 192/512/maskable/apple-touch + a vector `favicon.svg`; keeps the toolchain dependency-free and regenerable. **theme-color** is one dynamic `<meta id="theme-color">` updated by `applyTheme()` (+ the pre-paint inline script) rather than two media-query metas, because the app has a **manual** theme toggle that can disagree with the OS preference. **Offline read-only view = persisted Query cache, not SW runtime-caching of Supabase** — `PersistQueryClientProvider` + a localStorage `createSyncStoragePersister` (`gcTime` raised to 24 h ≥ `maxAge`; `staleTime` left at 0 so online behavior/Realtime are unchanged; only **successful** queries dehydrated). Chosen over caching raw authenticated API responses on disk (privacy on shared devices, and DELETE/RLS staleness); the cache is **wiped on sign-out** (`clearPersistedCache` in AuthProvider's `SIGNED_OUT`). **Queued offline writes deliberately skipped** ("if feasible" in the brief) — a correct write queue across optimistic caches + RLS conflicts is a large, error-prone surface; offline is read-only with a clear banner, writes fail-and-rollback (Phase 10+ candidate). **Reminders = "Both" (user-chosen):** the **reliable** path is an **email Edge Function on pg_cron**, the **zero-setup** path is **browser notifications** — they're complementary, not either/or, and neither adds frontend bundle weight (the function is server-side; notifications are ~the Notification API). **All reminder logic that needs `auth.users` (emails) or cross-user reads lives in two SECURITY DEFINER RPCs** (`due_reminder_candidates`, `mark_reminders_sent`) with **EXECUTE revoked from anon/authenticated and granted only to `service_role`**, so the browser can never enumerate emails; the Edge Function is additionally **secret-gated** (`x-cron-secret`) and deployed `--no-verify-jwt` (the cron, not a user, calls it). **Dedupe is `cards.reminder_sent_for date`** (the due_date last emailed) — `is distinct from due_date` re-arms a card automatically when its due date changes, no trigger needed. **Reminder secrets (`RESEND_API_KEY`/`REMINDER_FROM_EMAIL`/`CRON_SECRET`) are NOT `VITE_` vars** — they live only in Supabase Edge secrets (documented in `.env.example` + supabase/README). **Assignee picker added now** (was reserved since Phase 4) because reminders target a card's assignee: `AssigneeField` reuses the Phase 8 `useMembers` cache, and `assignee_id` threads through the **existing** `updateCardDetail` path so board **and** calendar persist it with no new mutation (same rationale as priority in the 2026-06-20 batch). `supabase/functions` + `scripts` added to ESLint ignores (Deno/Node, outside the app's typed graph).
- **2026-06-20** — **Phase 8 Collaboration (invitations, roles, realtime, presence).** **Invitations are keyed by email**, not user id, because the invitee may not have an account yet — an `invitations(project_id, email, role∈{editor,viewer}, invited_by)` row is **consumed into a `project_members` row** the moment that email becomes a user. Two redemption paths share one SECURITY DEFINER core (`redeem_invitations_for(user, email)`, a `delete…returning`→`insert…on conflict do nothing` atomic move): a **signup trigger** on `auth.users` auto-joins brand-new users, and a **`redeem_my_invitations()` RPC** (called from `ProjectsPage` on load) covers already-registered invitees. Redemption matches on the **authenticated user's own email on file** — the standard Supabase pattern; no email-ownership re-check (note for the security pass before charging). **Roles are enforced in RLS, not just the UI** (the brief's hard constraint): reads stay `is_project_member`, but every content **write** now gates on **`can_edit_project(project_id)`** (owner/editor) — and card-children on **`can_edit_card(card_id)`** — so viewers are read-only at the database; the UI threads a `canEdit` prop purely for polish (hidden compose/drag/edit affordances + a read-only card view + read-only notes). **The owner is sacrosanct:** ownership = `projects.owner_id`, and a `protect_project_owner_membership` BEFORE trigger forbids demoting/removing the owner's `project_members` row or granting `owner` to anyone else — with a **null-guard** (`projects` row already gone ⇒ it's a project-cascade delete ⇒ allow) so deleting a whole project still works. `profiles` got a second, additive SELECT policy via **`shares_a_project_with()`** so co-members can read each other's name/avatar (own-row-only would show ids). **Realtime = debounced cache invalidation, not surgical patching:** `useProjectRealtime` subscribes to `postgres_changes` for the active project and, on any event, **debounce-invalidates** the matching project-scoped TanStack key (board / card-extras / notes / members). Refetch is the authoritative, RLS-filtered snapshot → no duplicates, no flicker (TanStack swaps only on real change), conflict-tolerant, and **no cross-project leak** (every invalidation/refetch is keyed to this `projectId`; Realtime's own RLS limits which events even arrive). Chosen over per-event patching because DELETE payloads carry only the row identity and the card-child tables (`checklist_items`/`card_labels`) have no `project_id` to patch by — for those two we subscribe unfiltered and just refresh **this** project's extras (`REPLICA IDENTITY FULL` lets the project_id filters match on deletes for the rest). **Presence** uses Realtime Presence on a per-project channel keyed by user id (multi-tab collapses to one person) to show live viewers. **Members data is two queries, not a PostgREST embed** (`project_members.user_id`→`auth.users`, so there's no FK to embed `profiles` across), joined client-side. **Email is normalised to lowercase at the schema** so it matches the `lower(email)` unique index + redemption lookup with a plain equality filter.
- **2026-06-20** — **Phase 7 Notes/docs.** A `notes` row is **project content**, so it reuses the multi-tenant pattern **verbatim** — `project_id` + RLS on every op gated by **`is_project_member(project_id)`**, no new helper. **`updated_at` is server-owned:** a `before update` trigger (`touch_notes_updated_at`, `search_path=''`) stamps it on every write, so the list's "newest-edited-first" order and the "saved" time never trust the client (autosave fires many UPDATEs). **Autosave design:** the editor holds local title/content state, derives save status **during render** (no `setState` in effects — the repo's react-hooks lint forbids it), and **debounces 700 ms**; the actual write only runs inside the timeout / mutation callbacks. A **flush-on-unmount** effect (a ref-held closure, written in an effect — never during render) saves a pending edit when you switch notes inside the debounce window, so nothing is lost. Optimistic writes go through one `['notes', projectId]` cache (board/todos pattern); `mutate` is destructured for a **stable** reference so background re-renders can't reset the debounce timer. **Markdown is hand-rolled, on purpose:** `markdown.tsx` parses a common subset to **React elements** — XSS-safe by construction (React escapes text; no `dangerouslySetInnerHTML`; link hrefs allow-listed to http/https/mailto/relative) — chosen over a library because the brief said avoid heavy deps, the bundle is already over the warning, and a renderer (not a rich-text editor) is the smaller, safer surface. **Tabs are URL-driven** (`?tab=notes` via `useSearchParams`, replace) so the global `/notes` picker (`NotesHome`) can deep-link straight to a project's notes. **react-hooks lint gotcha (new this session):** the repo's ESLint flags (a) writing/reading a ref's `.current` during render and (b) synchronous `setState` inside an effect body — both must move to event handlers / async callbacks / effects; derive display state during render instead.
- **2026-06-20** — **Daily to-do planner = a private, non-project feature.** Lists/items belong to **one user**, not a project, so RLS is the simple **own-rows** pattern (like `profiles`), not membership. `todo_lists.user_id` **defaults to `auth.uid()`** so inserts omit it; `todo_items` gate through a new SECURITY DEFINER **`owns_todo_list(list_id)`** helper (resolves the list's owner once — same "keep policies flat" rationale as `can_access_card`). The model is **per-day, multi-list**: a `(list_date, name)` list holds items, so a day can have several named lists (Personal/Work). State is one cache **per day** (`['todos', dateKey]` → `{lists, items}`), optimistic like the board. No drag-reorder (append by fractional `position`); kept out of the project nav as its own sidebar tab + `/todos` route.
- **2026-06-20** — **Task priority is an open-ended integer, not an enum.** Per the request ("P1…P10 and beyond if more tasks"), `cards.priority` is a nullable `integer >= 1` (1 = most urgent, NULL = unset) — no fixed set, so big boards can use P11+. UI: `PriorityField` offers P1–P10 quick chips + a "Higher…" number input; `lib/priority.ts` is the single source of `formatPriority`/tier→color used by both the picker and the card-face pill. It **rides the existing `updateCardDetail`** path (added one column to the patch), so the board's `useUpdateCard` and the calendar's `useUpdateCalendarCard` both persist it with no new mutation. **Card-modal add bug (same day):** the checklist + label composers were `<form>`s nested in the modal's `<form>` — invalid HTML that made "+" submit the outer card form; the fix (here and the pattern going forward) is **no nested `<form>`s** — list composers use a div + click/Enter handlers.
- **2026-06-18** — **Phase 6 Calendar view:** No schema change — the calendar is a **read/rewrite over `cards.due_date`** (populated since Phase 5). **One query feeds the whole view:** `fetchDatedCards()` selects every card with a non-null `due_date` (RLS scopes it to the user's projects), and the **scope** (all-projects vs one project) is a client-side filter, so switching scope costs nothing. **Cross-cache mutations** are the key design: a card lives simultaneously in `['calendar-cards']`, its project's `['board', id]`, and `['card-extras', id]`, so reschedule/edit/delete patch **all relevant caches optimistically** (and invalidate them on settle) — drag a card to a new day here and it's already moved when you open the board. **Drag-to-reschedule** uses dnd-kit's low-level `useDraggable`/`useDroppable` (not Sortable — there's no intra-day order to persist): each chip is draggable, each `DayCell` is a droppable keyed by its `toDateKey` (`YYYY-MM-DD`), and drop sets `due_date` to that key. Only the in-grid chip registers a draggable for a given id; the lifted clone, agenda rows, and overflow-modal rows reuse the **presentational `CardChip`** (no hook) so there's never a duplicate-draggable-id. **Chips encode two signals at once:** project identity (accent gradient bar, via `accentVars`) and urgency (a danger/warning dot from the Phase 5 `due.ts`). **Clicking a chip opens the exact Phase 5 `CardDetailModal`** (imported from `features/board`), wired to the calendar's own mutations — full reuse, no fork. **Responsiveness** is a JS media-query split (`useMediaQuery('(min-width:640px)')`) rather than CSS `hidden`, specifically so the month grid's dnd droppables and the agenda don't both mount; mobile gets a tap-to-open agenda where rescheduling happens via the card's existing date picker. **Date math is all date-fns**, local-time throughout (`toDateKey`/`isToday`/`parseISO` all align with `due.ts`'s `startOfToday`), so a card sits on the same day it was set to regardless of timezone — no heavy calendar lib. **Filename gotcha:** the helpers file was first named `calendarGrid.ts`, which collides with the `CalendarGrid.tsx` component on case-insensitive Windows/macOS filesystems (tsc errors) — renamed to `dates.ts`. **Reduced-motion** gates the drag-overlay spring; the grid otherwise relies on the global reduced-motion CSS.
